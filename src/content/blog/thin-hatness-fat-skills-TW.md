Steve Yegge 說，使用 AI 程式代理（AI coding agents）的人「生產力是今天使用 Cursor 和聊天的工程師的 10 倍到 100 倍，大約也是 2005 年 Google 工程師的 1000 倍。」

這是真實數字；我看過，也親身經歷過。但人們聽到這件事時，通常會找錯解釋：更好的模型、更聰明的 Claude、更多參數。2 倍的人和 100 倍的人使用的是同一批模型；差異不在智慧，而在架構，而且可以寫在一張索引卡（index card）上。

## 外殼就是產品

2026 年 3 月 31 日，Anthropic 意外把 Claude Code 的完整原始碼發布到 npm 登錄檔（npm registry），共 512,000 行；我讀完後，它證實了我一直在 YC 教的事：祕密不在模型，而在包住模型的那個東西。

即時儲存庫脈絡（live repo context）、提示快取（prompt caching）、專用工具（purpose-built tools）、脈絡膨脹最小化（context bloat minimization）、結構化工作階段記憶（Structured session memory）、平行子代理（parallel sub-agents），這些都不會讓模型變得更聰明；它們的作用是讓模型在正確時間取得正確脈絡，而且不會被雜訊淹沒。

這個包裝層稱為外殼（harness）。每個 AI 建構者都應該問的問題是：哪些東西應該放進外殼，哪些東西應該留在外面？答案有特定形狀，我稱它為**薄外殼，厚技能（thin harness, fat skills）**。

## 五個定義

瓶頸從來不是模型的智慧。模型已經知道如何推理、綜合與撰寫程式碼；它們失敗，是因為不了解你的資料：你的結構描述（schema）、你的慣例、你的問題所具有的特定形狀。五個定義可以修正這件事。

### 1. 技能檔案

技能檔案（skill file）是一份可重複使用的 Markdown 文件，用來教模型如何做某件事；不是教它要做什麼，使用者會提供這一點；技能提供流程。

多數人漏掉的關鍵洞見是：**技能檔案的運作方式像方法呼叫（method call）**。它接收參數，你用不同引數呼叫它；同一套程序會依據你傳入的內容，產生截然不同的能力。

想像一個稱為 `/investigate` 的技能，它有七個步驟：界定資料集範圍、建立時間軸、為每份文件製作說話者與事件紀錄、綜合（synthesize）、為正反兩方論證、引用來源。它接收三個參數：`TARGET`、`QUESTION` 與 `DATASET`。把它指向一位安全科學家與 210 萬封揭露程序電子郵件，你會得到一位醫學研究分析師，判斷吹哨者是否被噤聲；把它指向一家空殼公司與 FEC 申報資料，你會得到一位鑑識調查員，追蹤協調式競選捐款。

同一個技能，同樣七個步驟，同一份 Markdown 檔案；技能描述的是判斷流程，呼叫時提供的是世界。

這不是提示工程（prompt engineering），而是軟體設計（software design），用 Markdown 作為程式語言，並以人類判斷作為執行環境（runtime）。事實上，Markdown 比僵硬的原始碼更適合封裝能力，因為它用模型原本就會思考的語言描述流程、判斷與脈絡。

### 2. 外殼

外殼是執行大型語言模型（LLM）的程式，它做四件事：以迴圈執行模型、讀寫你的檔案、管理脈絡、強制執行安全限制。就這樣，這就是「薄」。

反模式（anti-pattern）是厚外殼搭配薄技能。你看過這種東西：40 多個工具定義吃掉一半脈絡視窗；萬能工具透過 MCP 來回傳輸，每次花 2 到 5 秒；REST API 包裝器把每個端點都變成獨立工具；三倍的 token、三倍的延遲、三倍的失敗率。

你真正需要的是快速且窄的專用工具：一個 Playwright 命令列介面（CLI），每個瀏覽器操作用 100 毫秒完成，而不是一個 Chrome MCP，為了截圖、尋找、點擊、等待、讀取花 15 秒。這是 75 倍的差距。軟體現在不用再被過度珍惜；打造剛好需要的東西，讓範圍保持精準。

### 3. 解析器

解析器（resolver）是脈絡的路由表。當任務類型 X 出現時，先載入文件 Y。

技能告訴模型如何做，解析器告訴模型要載入什麼，以及何時載入。某位開發者修改提示；沒有解析器時，他們會直接發布；有解析器時，模型會先讀 `docs/EVALS.md`，其中寫著：執行評估套件、比較分數，若準確率下降超過 2%，就還原並調查。開發者原本不知道評估套件存在，解析器在正確時刻載入了正確脈絡。

Claude Code 內建解析器。每個技能都有描述欄位，模型會自動把使用者意圖對應到技能描述；你永遠不必記得 `/ship` 存在，描述本身就是解析器。

說實話：我的 CLAUDE.md 曾經有 20,000 行，每個怪癖、每個模式、每個我遇過的教訓都塞進去，完全荒謬。模型的注意力下降，Claude Code 真的叫我把它縮短。修正後大約 200 行，只放指向文件的指標；解析器會在重要時刻載入正確文件。兩萬行知識可以按需求存取，而且不污染脈絡視窗。

### 4. 潛在與確定性

系統中的每個步驟都屬於兩者之一，而混淆它們是代理設計中最常見的錯誤。

**潛在空間（latent space）** 是智慧存在的地方：模型閱讀、詮釋、決定，進行判斷、綜合與模式辨識。

**確定性（deterministic）** 是信任存在的地方：相同輸入，相同輸出，每一次都一樣，例如 SQL 查詢、編譯後的程式碼與算術。

大型語言模型可以安排 8 個人的晚餐座位，並考量個性與社交動態。請它安排 800 個人，它會幻覺出一張看似合理但完全錯誤的座位表。這是被迫放進潛在空間的確定性問題，也就是組合最佳化（combinatorial optimization）。最差的系統把錯誤工作放在線的錯誤一側；最好的系統會嚴格處理這條界線。

### 5. 說話者與事件紀錄

說話者與事件紀錄（diarization）是讓 AI 對真實知識工作有用的步驟。模型讀完某個主題的所有資料，然後寫出結構化側寫：從數十份或數百份文件濃縮出來的一頁判斷。

沒有 SQL 查詢能產生這個，也沒有 RAG 管線（RAG pipeline）能產生這個。模型必須真的閱讀，在心中同時保留矛盾，注意到什麼在何時發生變化，並綜合出結構化情報。這就是資料庫查詢與分析師簡報之間的差異。

## 架構

這五個概念組合成一個簡單的三層架構。

**厚技能**位於最上層：以 Markdown 撰寫的程序，編碼判斷、流程與領域知識。90% 的價值都在這裡。

**薄命令列外殼**位於中間：約 200 行程式碼，JSON 輸入，文字輸出，預設唯讀。

**你的應用程式**位於底層：QueryDB、ReadDoc、Search、Timeline，也就是確定性基礎。

這個原則有明確方向：把智慧往上推進技能，把執行往下推進確定性工具，維持外殼很薄。這樣做之後，模型的每次改進都會自動改善每個技能，而確定性層會維持完全可靠。

## 會學習的系統

我來示範這五個定義如何一起運作；不是理論，而是我們正在 YC 建構的實際系統。

Chase Center，2026 年 7 月，六千位創辦人在 Startup School。每個人都有結構化申請資料、問卷答案、1:1 顧問聊天逐字稿，以及公開訊號：X 貼文、GitHub 提交紀錄、顯示他們出貨速度的 Claude Code 逐字稿。

傳統做法是：一個 15 人的專案團隊閱讀申請資料，憑直覺判斷，更新試算表。它在 200 位創辦人時可行，到了 6,000 位就會失效。沒有人能在工作記憶中保留這麼多側寫，並注意到 AI 代理基礎設施小組的三位最佳候選人，分別是一位在 Lagos 的開發工具創辦人、一位在 Singapore 的法規遵循創辦人，以及一位在 Brooklyn 的命令列工具創辦人，而他們都在 1:1 聊天中用不同詞彙描述同一個痛點。

模型可以做到，做法如下。

**增補（Enrichment）**：一個稱為 `/enrich-founder` 的技能會拉取所有來源、執行增補、製作說話者與事件紀錄，並標示創辦人所說內容與實際建構內容之間的落差。確定性層處理 SQL 查詢、GitHub 統計、展示網址的瀏覽器測試、社群訊號擷取、CrustData 查詢；排程工作每晚執行，六千份側寫永遠保持最新。

說話者與事件紀錄輸出會抓到關鍵字搜尋找不到的東西：

FOUNDER: Maria Santos\
COMPANY: Contrail (contrail.dev)\
SAYS: "Datadog for AI agents"\
ACTUALLY BUILDING: 80% of commits are in billing module.\
她正在打造一個偽裝成可觀測性工具的 FinOps 工具。

那個落差，也就是「所說」與「實際建構」之間的落差，需要閱讀 GitHub 提交歷史、申請資料與顧問逐字稿，並同時把三者放在心中。沒有嵌入相似度搜尋能找到這一點，也沒有關鍵字篩選器能找到它；模型必須閱讀完整側寫並做出判斷。（這是最適合放進潛在空間的決策！）

**配對（Matching）**：這就是技能即方法呼叫（skill-as-method-call）發光的地方。同一個配對技能呼叫三次，產生三種完全不同的策略：

`/match-breakout` 接收 1,200 位創辦人，依產業親近性分群，每間房 30 人，使用嵌入加上確定性分派；`/match-lunch` 接收 600 人，跨產業做意外性配對，每桌 8 人，不重複；大型語言模型發明主題，接著由確定性演算法分配座位；`/match-live` 處理此刻人在場館內的所有人，使用最近鄰嵌入（nearest-neighbor embedding），200 毫秒，1:1 配對，排除已經見過面的人。

而模型能做出分群演算法永遠無法做出的判斷：「Santos 和 Oram 都是 AI 基礎設施，但他們不是競爭對手；Santos 做的是成本歸因，Oram 做的是協調編排。把他們放在同一組。」或者：「Kim 申請時填的是『開發者工具』，但他的 1:1 逐字稿顯示他正在打造 SOC2 法規遵循自動化；把他移到 `FinTech/RegTech`。」

沒有任何嵌入能捕捉 Kim 的重新分類。模型必須閱讀完整側寫。

**學習迴圈（learning loop）**：活動結束後，`/improve` 技能會讀取 NPS 問卷，為普通回覆製作說話者與事件紀錄；不是糟糕回覆，而是「還可以」的回覆，也就是系統差一點成功但沒有成功的地方；接著萃取模式，提出新規則，並把規則寫回配對技能：

When attendee says "AI infrastructure"\
    but startup is 80%+ billing code:\
    → Classify as FinTech, not AI Infra.\

When two attendees in same group\
    already know each other:\
    → Penalize proximity.\
       Prioritize novel introductions.

這些規則會寫回技能檔案；下一次執行會自動使用它們，技能會改寫自己。

7 月活動：12% 評分「還可以」；下一場活動：4%。技能檔案學會了「還可以」真正代表什麼，系統也在沒有人重寫程式碼的情況下變得更好。

同樣模式可以轉移到各處：*擷取、閱讀、製作說話者與事件紀錄、計數、綜合；然後，調查問卷、探查、製作說話者與事件紀錄、改寫技能。*

若你想知道 2026 年最有價值的迴圈是什麼，就是這些。我們可以把它們套用到所有知識工作的學科與人生領域。

## 技能是永久升級

我最近在 X 上貼了一段給 OpenClaw 的指令，得到的共鳴比我預期更多：

> 你不可以做一次性工作。如果我要求你做某件事，而且它屬於未來還會再次發生的事情，你必須：第一次先手動處理 3 到 10 個項目，把輸出給我看；若我核准，就把它編碼成技能檔案；若它應該自動執行，就把它放到排程工作上。測試標準：如果我必須為同一件事問你兩次，你就失敗了。

一千個讚與兩千五百個書籤。人們以為這是提示工程技巧；其實不是，它是我一直在描述的架構。你寫下的每個技能，都是對系統的永久升級；它永遠不會退化，永遠不會忘記，會在凌晨 3 點趁你睡覺時執行。而當下一個模型發布時，每個技能都會立刻變得更好：潛在步驟中的判斷會改善，確定性步驟則維持完全可靠。

這就是取得 Yegge 所說 100 倍的方式。不是更聰明的模型，而是厚技能、薄外殼，以及把所有事情編碼成技能的紀律。

系統會複利成長；建構一次，永久執行。
