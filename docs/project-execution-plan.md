# 個人品牌網站最終版執行計畫

## 目前狀態

目前專案已完成第一版個人品牌網站的技術基底：

- 使用 Astro 建立靜態網站。
- 使用 Markdown 維護文章與專案內容。
- 建立首頁、關於頁、專案列表、專案詳細頁、文章列表與文章頁。
- 使用內容集合管理部落格與專案資料。
- 新增 GitHub Actions 部署流程，推送到 `main` 後可建置並發布到 GitHub Pages。
- 新增 `npm run verify`，用來檢查建置輸出、草稿排除與主要頁面預覽。

目前仍屬於第一版骨架：

- 網站內容多數仍是佔位文字。
- 專案圖片與主視覺仍是暫用圖片。
- 個人品牌定位、正式姓名、職稱、經歷、作品成果與聯絡資訊尚未補齊。
- 視覺設計仍是初步版本，尚未定稿。
- Markdown 編輯方式可用，但尚未達到接近 HackMD 的低摩擦管理體驗。

## 核心目標

最終版網站要達成以下目標：

- 內容可用 Markdown 維護，讓文章、專案與工作紀錄能低成本更新。
- 專案可用簡易圖文展示，讓讀者快速理解問題、做法、技術與成果。
- 網站可作為部落格，支援長期累積技術文章、專案復盤與工作筆記。
- UI 與功能維持分離，修改視覺設計時不影響內容模型、路由與草稿排除機制。
- 網站能建立工作上的個人品牌，讓合作對象、招募者與技術社群快速認識網站擁有者。

## 第 1 階段：個人品牌與內容策略

### 目標

先定義網站要傳達的身份與訊息，再補正式內容。這一階段決定網站的敘事方向。

### 待實作內容

- 決定正式姓名、職稱、定位語與一句話介紹。
- 補上正式聯絡信箱、GitHub、LinkedIn 或其他公開社群連結。
- 規劃首頁主訊息，讓訪客在短時間內理解：
  - 你是誰。
  - 你的專長是什麼。
  - 你做過哪些代表性成果。
  - 訪客下一步可以看什麼或如何聯絡你。
- 規劃關於頁敘事，包含背景、工作方式、技術興趣與職涯方向。
- 規劃專案分類，例如網站開發、資料分析、工具開發、研究紀錄或其他合適分類。
- 規劃文章主題，例如技術筆記、專案復盤、學習紀錄、工作方法。

### 產出

- 正式首頁文案草稿。
- 正式關於頁文案草稿。
- 代表專案清單。
- 文章主題清單。

## 第 2 階段：資訊架構與內容模型

### 目標

讓內容資料結構足以支援最終版展示需求，同時維持 Markdown 維護方式。

### 待實作內容

- 擴充專案欄位，支援：
  - 專案角色。
  - 專案時間。
  - 成果摘要。
  - 專案分類。
  - 技術分類。
  - 截圖或展示圖片。
  - 案例研究段落。
- 擴充文章欄位，支援：
  - 文章分類。
  - 系列文章。
  - 更新日期。
  - 社群預覽摘要。
- 評估是否新增以下頁面：
  - 履歷頁。
  - 經歷頁。
  - 技能頁。
  - 聯絡頁。
  - 履歷下載入口。
- 調整內容集合定義，確保欄位有明確驗證規則。

### 產出

- 更新後的文章 Markdown 模板。
- 更新後的專案 Markdown 模板。
- 最終版網站頁面清單。
- 內容欄位說明文件。

## 第 3 階段：整體視覺設計

### 目標

重新定義網站視覺，使其更符合工作上的個人品牌，而不是只停留在第一版骨架。

### 待實作內容

- 定義色彩系統，包含背景、文字、主色、輔助色、邊框與狀態色。
- 定義字體與排版規則，包含標題、內文、列表、程式碼與引用區塊。
- 設計首頁主視覺，讓主訊息、代表專案與行動入口更明確。
- 重設專案卡片，讓封面、摘要、技術標籤與成果資訊更清楚。
- 重設文章列表，強化日期、分類、標籤與摘要層次。
- 建立設計系統規則，至少包含：
  - 按鈕。
  - 標籤。
  - 卡片。
  - 導覽列。
  - 頁尾。
  - 內容頁排版。
- 保留目前 Astro 元件分層，讓設計調整集中在元件與樣式中，內容仍由 Markdown 管理。

### 產出

- 最終版視覺方向。
- 可重用 UI 元件規則。
- 首頁、專案頁、文章頁與關於頁的新版版面。

## 第 4 階段：Markdown 編輯體驗

### 目標

讓新增文章與專案的流程接近低摩擦 Markdown 編輯，降低後續維護成本。

### 待實作內容

- 在 README 與文件中補齊文章模板、專案模板與圖片放置規則。
- 建立更完整的文章範例，讓新文章可以直接複製。
- 建立更完整的專案範例，讓新專案可以直接複製。
- 補充圖片命名規則，例如專案封面、內文截圖與個人照片。
- 補充本機預覽流程，說明 `npm run dev`、`npm run build`、`npm run verify` 的用途。
- 評估第二階段導入內容管理系統，例如 Decap CMS 或 TinaCMS。
- CMS 導入前，先保留 GitHub 網頁編輯器、VS Code、Obsidian 等 Markdown 編輯方式。

### 產出

- Markdown 內容維護指南。
- 文章模板。
- 專案模板。
- 圖片管理規則。
- CMS 導入評估結論。

## 第 5 階段：網站功能完整度

### 目標

補齊正式網站常見的可用性、搜尋與分享能力。

### 待實作內容

- 新增 SEO 基本設定：
  - 每頁標題。
  - 每頁描述。
  - 社群預覽圖片。
  - Canonical URL。
- 新增站點地圖。
- 新增 RSS，讓文章可被訂閱。
- 新增專案篩選或專案標籤頁，讓專案數量增加後仍好瀏覽。
- 新增文章標籤列表、文章分類或系列文章入口。
- 新增 404 頁。
- 補強空狀態，例如沒有文章或沒有專案時的顯示內容。
- 統一外部連結樣式與開啟方式。
- 補強可及性，包含鍵盤導覽、連結文字、圖片替代文字與色彩對比。

### 產出

- 可發布的 SEO 設定。
- 站點地圖與 RSS。
- 專案篩選或標籤入口。
- 文章分類或標籤入口。
- 404 頁與空狀態設計。

## 第 6 階段：品質驗證與發布

### 目標

確保最終版在本機與 GitHub Pages 上都能穩定運作。

### 待實作內容

- 每個階段完成後執行：

```bash
npm run build
npm run verify
```

- 檢查手機、平板與桌面版面。
- 檢查文字是否溢出。
- 檢查專案圖片比例是否一致。
- 檢查導覽列、按鈕、專案連結與文章連結。
- 檢查鍵盤導覽。
- 檢查色彩對比。
- 推送到 `main` 後確認 GitHub Actions 建置結果。
- 確認 GitHub Pages 正式網址能瀏覽最新版本。

### 產出

- 通過本機建置與驗證。
- 通過主要頁面人工檢查。
- GitHub Actions 建置成功。
- GitHub Pages 發布成功。

## 驗收標準

### 內容層面

- 首頁具備正式個人品牌文案。
- 關於頁具備可公開的正式敘事。
- 至少 3 個正式專案具備可公開內容。
- 至少 3 篇正式文章具備可公開內容。
- 所有文章與專案都能只透過 Markdown 更新。

### 設計層面

- 首頁能在 10 秒內讓訪客理解網站擁有者的身份、專長與代表成果。
- 手機與桌面都能穩定閱讀。
- 文字不溢出。
- 專案圖片比例一致。
- 導覽、卡片、標籤、按鈕與內容頁排版一致。

### 技術層面

- `npm run build` 成功。
- `npm run verify` 成功。
- 草稿文章與草稿專案不會出現在正式頁面。
- GitHub Actions 成功建置。
- GitHub Pages 可正常瀏覽最新版本。

## 建議實作順序

1. 先完成個人品牌與內容策略，避免過早調整 UI。
2. 再擴充內容模型，讓專案與文章能承載正式內容。
3. 接著重設視覺設計，讓網站從第一版骨架轉為正式作品集。
4. 補齊 Markdown 編輯文件，降低後續維護成本。
5. 補上 SEO、RSS、站點地圖、篩選與 404 等正式網站功能。
6. 完成跨裝置檢查、本機驗證與 GitHub Pages 發布驗證。

## 暫不納入第一輪最終版的項目

- 登入功能。
- 後端資料庫。
- 即時協作編輯。
- 複雜動畫或大型互動效果。
- 電子報訂閱系統。
- 訪客留言系統。

這些項目可在網站內容與品牌定位穩定後再評估。

## 假設

- 最終版仍以 Astro、Markdown 與 GitHub Pages 為核心架構。
- 目前第一版網站作為技術基底保留，後續以階段方式逐步替換內容與設計。
- 第一輪最終版優先處理品牌表達、內容管理、作品展示與部落格能力。
- 線上內容管理系統會在 Markdown 工作流程穩定後再決定是否導入。
