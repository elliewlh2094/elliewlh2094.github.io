# 技術疑難排解指南

這份文件記錄本專案維護過程中已遇到的技術問題，包含症狀、常見原因、處理方式與預防規則。遇到建置、驗證、內容集合、草稿排除、封面圖片或長文章展示頁異常時，先依照本文件檢查。

相關文件分工：

- 日常操作流程、內容模板與指令說明，查看 `README.md`。
- 第一版目前狀態、優先順序與範圍控制，查看 `docs/project-execution-plan.md`。
- 維護規則、內容模型與修改慣例，查看 `AGENTS.md`。

## 處理問題前的檢查清單

- 先執行 `git status --short`，確認目前有哪些未提交修改。
- 修改前先讀取目標檔案，確認使用者或前一次工作已經留下的內容。
- 只修改和目前任務直接相關的檔案與區塊。
- 不要還原未確認來源的變更；遇到同一檔案已有其他修改時，沿用現有內容再做最小調整。
- 修改內容、頁面、元件或樣式後，依照影響範圍執行 `npm run build` 或 `npm run verify`。
- 只修改文件時，確認 Markdown 格式、路徑與專案事實一致即可。

## 內容集合欄位變更後建置失敗

### 症狀

- `npm run build` 顯示內容集合（content collections）驗證錯誤。
- 錯誤訊息指出某篇 Markdown 前置資料缺少欄位，或包含內容模型已移除的欄位。
- 新增或移除 `src/content.config.ts` 欄位後，正式文章、專案或草稿其中一部分無法通過建置。

### 常見原因

- `src/content.config.ts` 已更新，`src/content/blog` 或 `src/content/projects` 內的 Markdown 前置資料尚未同步。
- 只更新正式內容，遺漏 `draft: true` 的草稿內容。
- 文件或範例模板仍保留舊欄位，導致後續新增內容沿用錯誤格式。

### 處理方式

- 先檢查 `src/content.config.ts`，確認目前實際欄位定義。
- 搜尋被移除或新增的欄位名稱，例如 `updatedDate`、`category`、`cover`、`featured`、`summary`。
- 同步更新正式內容與草稿內容。
- 同步更新 `README.md` 或相關文件中的內容模板。
- 執行 `npm run build`，確認所有內容集合資料都能通過驗證。

### 預防規則

- 內容模型變更要同時更新三個地方：結構定義、所有內容檔、維護文件。
- 草稿仍會參與內容集合驗證，因此欄位遷移也要包含草稿。

## 草稿排除與 verify 檢查失敗

### 症狀

- `npm run verify` 找不到預期的正式頁面。
- 專案或文章列表卡片數量和預期不同。
- `dist` 中沒有某篇文章或專案的詳細頁。
- `npm run verify` 顯示缺少公開內容封面資源。

### 常見原因

- 內容檔被設定為 `draft: true`。
- `getStaticPaths` 或列表頁過濾掉草稿內容。
- 公開內容的 `cover` 指向了不存在的公開圖片路徑。
- `scripts/verify-site.mjs` 會依目前 `draft: false` 的內容動態檢查正式輸出，和草稿展示需求不同。

### 處理方式

- 檢查目標 Markdown 的 `draft` 欄位。
- 正式公開內容使用 `draft: false`。
- 只用於排版測試的內容保持 `draft: true`，並透過 `/longform-demo/` 這類展示頁檢查。
- 若預期正式輸出包含該頁，執行 `npm run build` 後檢查 `dist` 中是否產生對應路徑。
- 若公開內容有 `cover`，同步檢查 `public/images` 中是否存在對應圖片，並確認建置後的 `dist/images/...` 已輸出。
- 公開內容增減、草稿狀態變更時，優先重新執行 `npm run build` 與 `npm run verify`，不需要先改驗證腳本。

### 預防規則

- 正式列表與正式詳細頁應排除草稿。
- 草稿展示需求應使用獨立展示頁，不要改動正式列表邏輯。
- 公開內容路徑應由 Markdown 檔名與相對目錄穩定對應，避免隨意改名造成輸出路徑變動。
- 更新驗證腳本前，先確認變更的是驗證規則，而不是單純的公開內容清單。

## Astro 快取與重複識別碼異常

### 症狀

- Astro 顯示 `Duplicate id` 相關警告。
- 內容頁數或輸出結果和 Markdown 檔案狀態不一致。
- 修改或搬移內容檔後，建置結果仍像使用舊資料。

### 常見原因

- Astro 快取保留了舊的內容集合資料。
- 檔名、slug 或內容集合識別碼短時間內多次變更。
- 建置輸出或本機開發伺服器尚未重新整理。

### 處理方式

- 停止本機開發伺服器後重新啟動。
- 重新執行 `npm run build`。
- 檢查 `src/content/blog` 與 `src/content/projects` 是否有重複檔名或重複 slug。
- 若快取仍造成異常，可清除 Astro 產生的快取資料後再建置。

### 預防規則

- 新增內容時使用明確且唯一的檔名。
- 重新命名內容檔後，檢查舊檔是否仍存在。
- 同一批修改中避免反覆建立、刪除、重新命名同一篇內容。

## 封面圖片被裁切或顯示規則錯誤

### 症狀

- 專案或部落格詳細頁的封面圖片被裁切，只顯示局部。
- 卡片封面正常，但文章頁封面比例不正確。
- 沒有 `cover` 的部落格文章仍顯示不需要的格線區塊。
- 封面區重複顯示標題。

### 常見原因

- 卡片封面和詳細頁封面共用同一組 `object-fit: cover` 樣式。
- 詳細頁需要完整顯示圖片，但卡片需要固定比例裁切。
- 有 `cover` 和沒有 `cover` 的部落格文章顯示邏輯不同。
- 占位格線區塊仍保留舊的標題文字。

### 處理方式

- 卡片封面維持固定比例與 `object-fit: cover`。
- 詳細頁封面使用完整圖片顯示，搭配 `object-fit: contain` 或移除固定高度裁切。
- 部落格詳細頁只有在文章有 `cover` 時顯示封面圖片。
- 部落格卡片沒有 `cover` 時可顯示格線占位，但占位區不顯示標題。
- 專案卡片與部落格卡片的占位規則分開確認，避免互相影響。

### 預防規則

- 修改封面樣式時，分別檢查卡片、專案詳細頁、部落格詳細頁。
- 圖片顯示需求分成兩類：卡片可裁切以維持一致比例，文章頁以完整顯示為主。
- 新增 `cover` 欄位後，同步檢查有圖片與無圖片兩種狀態。

## 長文章展示頁顯示異常

### 症狀

- `/longform-demo/` 無法顯示草稿範例。
- 長程式碼區塊在手機寬度撐破版面。
- 圖片、圖片說明、引用區塊或程式碼區塊樣式和正式文章不同。
- 展示頁出現在導覽列或被搜尋引擎索引。

### 常見原因

- 展示頁沒有直接讀取草稿內容。
- 長文章樣式沒有沿用 `.content-page`、`.content-hero`、`.prose`、`.project-summary`。
- `BaseLayout.astro` 沒有設定 `noindex` 支援，或展示頁沒有啟用。
- 媒體區塊寬度與文字欄寬度沒有分開設定。

### 處理方式

- 確認 `/longform-demo/` 直接渲染指定草稿內容。
- 展示頁不要加入導覽列與首頁入口。
- 展示頁設定 `noindex: true`，頁首輸出 `robots` 中繼標籤。
- 檢查 `.prose pre`、`.prose figure`、`.prose > img`、`.content-cover`、`.project-summary` 的寬度與置中規則。
- 在桌面與手機寬度人工檢查程式碼區塊、圖片、圖片說明與引用區塊。

### 預防規則

- 長文章樣式調整後，同步檢查正式文章、正式專案與 `/longform-demo/`。
- 媒體區塊使用較窄寬度，文字欄可使用較寬寬度。
- 程式碼區塊需要保留水平捲動或換行策略，避免小螢幕溢出。

## 工作樹已有大量未提交修改

### 症狀

- `git status --short` 顯示多個檔案已修改。
- 同一檔案同時包含本次任務相關修改與其他既有修改。
- 修改範圍不清楚，容易覆蓋使用者剛完成的內容。

### 常見原因

- 多次任務累積在同一個工作樹中。
- 使用者在編輯器中同步修改檔案。
- 先前產生的檔案、圖片或樣式尚未提交。

### 處理方式

- 開始修改前先執行 `git status --short`。
- 對要修改的檔案先讀取目前內容。
- 使用局部補丁，只改任務指定段落。
- 遇到同一區塊已有不明修改時，先依照現有內容整合。
- 完成後用 `git diff -- <path>` 檢查本次變更範圍。

### 預防規則

- 不使用會整批重寫無關內容的格式化或產生指令。
- 不還原使用者或其他工作留下的變更。
- 文件、內容、樣式與元件分批處理，降低修改範圍重疊。
