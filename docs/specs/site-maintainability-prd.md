# 第二階段維護優先重整 PRD

## 目標

第二階段以「維護優先的內部整理」為主軸，整理目前 Astro 網站的共用查詢、頁面責任、中繼資料與樣式分層，降低未來新增內容或調整頁面時需要跨多個檔案同步修改的風險。

這一輪不追求新功能擴張，也不依賴大量內容更新。主要成果是建立穩定的程式骨架與對應維護文件。

## 使用者與價值

- 主要使用者：目前與未來維護這個網站的你自己。
- 次要使用者：後續接手這個專案的代理或工程師。
- 核心價值：
  - 集中發布、排序與精選規則。
  - 降低頁面模板中的資料處理負擔。
  - 讓樣式定位更快。
  - 保持內容模型的小幅相容式演進。

## 專案事實

### 技術棧

- Astro 5
- Markdown 內容集合
- GitHub Pages 靜態部署

### 指令

- 開發：`npm run dev`
- 建置：`npm run build`
- 預覽：`npm run preview`
- 驗證：`npm run verify`

### 目錄責任

- `src/content/`：文章與專案內容
- `src/lib/`：共用查詢、站點設定、顯示資料整理
- `src/components/`：畫面元件
- `src/layouts/`：版型與頁面殼層
- `src/pages/`：路由頁面
- `src/styles/`：樣式入口與分層模組
- `docs/`：維護文件與規格

## 需求

### 1. 內容查詢層

- 建立文章與專案的共用查詢函式。
- 集中處理發布條件、排序與精選規則。
- 詳細頁的 `getStaticPaths` 要與列表頁共用同一套發布條件。

### 2. 顯示資料與責任邊界

- 頁面負責組裝資料。
- 元件只接收顯示所需資料，不直接承擔集合規則。
- 中繼資料、日期格式與封面替代文字要由共用工具集中處理。

### 3. 站點設定與中繼資料

- 站名、預設描述、導覽項目、頁尾資訊集中管理。
- 頁面標題與 Open Graph 標題邏輯集中管理。

### 4. 樣式分層

- 保留 `src/styles/global.css` 作為單一入口。
- 將樣式拆成少數明確模組，至少包含：
  - 基礎
  - 版型
  - 元件
  - 內容頁

### 5. 最小內容模型整理

- `blog` 與 `projects` 支援可選 `coverAlt`。
- `projects` 支援 `tags`，預設為空陣列。
- 所有欄位變更都採相容式新增，不要求重寫既有 Markdown。

### 6. 文件

- 新增一份產品需求文件。
- 新增一份工作分解計畫。
- 新增一份未來優化方向文件。
- 更新 `README.md` 與 `AGENTS.md`，使其符合新結構。

## 程式風格

- 延用 Astro 現有 `.astro` 元件寫法。
- 只抽出已存在於多個頁面的重複邏輯。
- 共用函式以高訊號、低抽象為原則。

範例：

```ts
export async function getFeaturedProjects(limit = 3) {
  const projects = await getPublishedProjects();
  return projects.filter((project) => project.data.featured).slice(0, limit);
}
```

## 驗證策略

- 每個主要階段至少執行一次 `npm run build`。
- 涉及公開輸出、草稿排除或封面資源時執行 `npm run verify`。
- 樣式拆分後人工檢查：
  - 首頁
  - 關於頁
  - 文章列表
  - 專案列表
  - 文章詳細頁
  - 專案詳細頁
  - `longform-demo`

## 邊界

### Always

- 保持公開路由不變。
- 保持草稿排除規則不變。
- 每次整理後執行對應建置與驗證。
- 同步更新維護文件。

### Ask First

- 新增相依套件。
- 變更部署流程。
- 調整驗證腳本的目標範圍。
- 重新定義網站定位與內容方向。

### Never

- 修改 `node_modules`
- 取消草稿排除
- 為了推測性需求加入大型抽象層
- 把 `dist` 當作主要編輯來源

## 成功標準

- 首頁、列表頁與詳細頁不再各自重寫同一套發布與排序規則。
- 元件不再直接承擔集合邏輯。
- 樣式規則可以快速定位到對應模組。
- `npm run build` 與 `npm run verify` 通過。
- 文件能反映新的維護方式與後續方向。
